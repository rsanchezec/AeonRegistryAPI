Guia paso a paso para iniciar una API con autenticacion

Esta guia resume la base comun para proyectos tipo backend API: usuarios, login, roles, base de datos, tokens Bearer, Swagger y endpoints protegidos. El tema del negocio cambia, pero esta preparacion suele repetirse.

1. Vision General

En proyectos como inventarios, CRM, sistemas medicos, cursos, reportes o catalogos internos, casi siempre necesitas una base comun antes de crear las entidades propias del negocio.

Parte Para que sirve
ASP.NET Core Minimal API Define endpoints HTTP como GET, POST, PUT y DELETE.
Entity Framework Core Conecta C# con la base de datos usando modelos y migraciones.
ASP.NET Identity Maneja usuarios, passwords, roles, tokens y tablas de autenticacion.
JWT / Bearer token Permite llamar endpoints privados despues de iniciar sesion.
Swagger Permite documentar y probar la API desde el navegador.

2. Flujo De Trabajo

  1. Crear el proyecto API.
  2. Instalar paquetes de EF Core, PostgreSQL o SQL Server, Identity y Swagger.
  3. Crear el modelo de usuario personalizado.
  4. Crear el DbContext heredando de IdentityDbContext.
  5. Configurar la cadena de conexion.
  6. Registrar servicios en Program.cs.
  7. Configurar autenticacion y autorizacion.
  8. Crear migraciones y actualizar la base de datos.
  9. Probar register/login en Swagger.
  10. Crear services para logica reutilizable o integraciones externas.
  11. Crear middleware solo si necesitas interceptar requests globalmente.
  12. Crear entidades y endpoints propios del negocio.
  13. Proteger endpoints con RequireAuthorization() o policies.
En tu proyecto actual ya estas alrededor del paso 9: la base de Identity, tokens, Swagger y base de datos ya esta levantada.

3. Paquetes Necesarios

Para una API con Identity y PostgreSQL, normalmente instalas paquetes como estos:

dotnet add package Microsoft.AspNetCore.Identity.EntityFrameworkCore
dotnet add package Microsoft.EntityFrameworkCore
dotnet add package Microsoft.EntityFrameworkCore.Tools
dotnet add package Npgsql.EntityFrameworkCore.PostgreSQL
dotnet add package Swashbuckle.AspNetCore

Si el proyecto usa JWT explicito, tambien puede incluir:

dotnet add package Microsoft.AspNetCore.Authentication.JwtBearer
Paquete Uso
Identity.EntityFrameworkCore Permite guardar usuarios, roles y claims de Identity en la base de datos.
EntityFrameworkCore.Tools Habilita comandos como dotnet ef migrations add.
Npgsql.EntityFrameworkCore.PostgreSQL Proveedor para conectarse a PostgreSQL.
Swashbuckle.AspNetCore Genera Swagger UI y documentacion OpenAPI.

5. Clases Base

ApplicationUser

Es tu usuario personalizado. Hereda de IdentityUser, que ya trae campos como Id, Email, UserName, PasswordHash y seguridad. Aqui agregas campos propios como nombre, apellido, telefono adicional o estado. 

public class ApplicationUser : IdentityUser
{
    [Required]
    public string? FirstName { get; set; }

    [Required]
    public string? LastName { get; set; }

    public string FullName => $"{FirstName} {LastName}";
}

ApplicationDbContext

Es la clase que representa la base de datos desde C#. Si usas Identity, debe heredar de IdentityDbContext<ApplicationUser>. 

public class ApplicationDbContext
    : IdentityDbContext<ApplicationUser>
{
    public ApplicationDbContext(DbContextOptions<ApplicationDbContext> options)
        : base(options)
    {
    }

    public DbSet<Product> Products => Set<Product>();
}

Requests y Responses

Los modelos de request describen lo que entra a la API. Los modelos response describen lo que sale. Esto evita exponer directamente datos sensibles como PasswordHash. 

public class RegisterUserRequest
{
    [Required]
    public string? Email { get; set; }

    [Required]
    public string? FirstName { get; set; }

    [Required]
    public string? LastName { get; set; }
}

6. Program.cs

Program.cs es el centro de arranque. Ahi registras servicios, base de datos, autenticacion, autorizacion, Swagger y endpoints.

Orden comun

  1. Crear el builder.
  2. Registrar Swagger.
  3. Registrar DbContext.
  4. Registrar Identity.
  5. Registrar policies y servicios.
  6. Construir la app.
  7. Activar middleware: HTTPS, static files, auth.
  8. Mapear endpoints.
  9. Ejecutar la app.
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddcustomSwagger();

builder.Services.AddDbContext<ApplicationDbContext>(options =>
    options.UseNpgsql(connectionString));

builder.Services.AddIdentityApiEndpoints<ApplicationUser>(options =>
{
    options.SignIn.RequireConfirmedAccount = false;
})
.AddRoles<IdentityRole>()
.AddEntityFrameworkStores<ApplicationDbContext>();

builder.Services.AddAuthorizationBuilder()
    .AddPolicy("AdminOnly", policy => policy.RequireRole("Admin"));

var app = builder.Build();

app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseAuthentication();
app.UseAuthorization();

app.MapGroup("/api/auth").MapIdentityApi<ApplicationUser>();
app.MapCustomIdentityEndpoints();

app.Run();

7. Identity Y Usuarios

ASP.NET Identity crea y maneja tablas como AspNetUsers, AspNetRoles, AspNetUserRoles, AspNetUserClaims y mas. No tienes que crear esas tablas a mano.

Registro: crea un usuario y guarda el password como hash, nunca como texto plano.

Login: valida usuario y password, y devuelve token o cookie segun configuracion.

Roles: permiten dar permisos como Admin, User, Researcher.

Crear usuario desde codigo

var user = new ApplicationUser
{
    UserName = dto.Email,
    Email = dto.Email,
    FirstName = dto.FirstName,
    LastName = dto.LastName
};

var result = await userManager.CreateAsync(user, "Admin123!");
La password temporal debe cumplir las reglas de Identity: mayuscula, numero, simbolo y longitud suficiente, segun la configuracion del proyecto.

8. Tokens Y Login

Cuando haces login correctamente, Identity devuelve un accessToken. Ese token se manda luego en el header Authorization para llamar endpoints protegidos.

Login body

{
  "email": "user@email.com",
  "password": "Admin123!"
}

Header para endpoints protegidos

Authorization: Bearer TU_ACCESS_TOKEN

El token representa al usuario autenticado. Si el endpoint exige un rol, el usuario tambien debe tener ese rol asignado.

9. Swagger

Swagger permite probar la API desde el navegador. Para proyectos con tokens, conviene agregar seguridad Bearer para poder pegar el token y probar endpoints privados. 

c.AddSecurityDefinition("Bearer", new OpenApiSecurityScheme
{
    Name = "Authorization",
    Type = SecuritySchemeType.Http,
    Scheme = "bearer",
    BearerFormat = "JWT",
    In = ParameterLocation.Header
});
Si Swagger dice Failed to fetch, muchas veces no es error del login. Puede ser certificado HTTPS, puerto incorrecto, CORS o servidor apagado.

10. Migraciones

Las migraciones convierten tus clases C# en cambios reales de base de datos. Con Identity, la primera migracion suele crear las tablas AspNet.... 

dotnet ef migrations add InitialIdentity
dotnet ef database update

Cuando agregues una entidad nueva, por ejemplo Artifact, haces otra migracion:

dotnet ef migrations add AddArtifacts
dotnet ef database update

11. Endpoints Protegidos

Una vez que Identity funciona, puedes crear endpoints del negocio y protegerlos.

Endpoint publico

group.MapGet("/public", () =>
{
    return Results.Ok("Cualquiera puede ver esto");
});

Endpoint con login requerido

group.MapGet("/profile", (ClaimsPrincipal user) =>
{
    return Results.Ok(user.Identity?.Name);
})
.RequireAuthorization();

Endpoint solo admin

group.MapDelete("/{id}", async (int id, ApplicationDbContext db) =>
{
    // borrar registro
    return Results.NoContent();
})
.RequireAuthorization("AdminOnly");

12. Modulos Endpoint

En Minimal APIs, los endpoints pueden crecer rapido si dejas todo en Program.cs. Por eso conviene moverlos a clases por modulo: auth, productos, usuarios, reportes, archivos, etc.

Para que sirve un archivo endpoint

Un endpoint define la entrada HTTP de tu sistema. Su trabajo principal es recibir el request, validar lo basico, llamar servicios o base de datos, y devolver una respuesta HTTP.

Endpoint Responsabilidad
AuthEndpoints.cs Registro, login personalizado, forgot password, reset password, perfil.
ProductEndpoints.cs Crear, listar, actualizar y borrar productos.
ArtifactEndpoints.cs Endpoints propios del dominio del proyecto.
AdminEndpoints.cs Operaciones administrativas protegidas por rol.

Patron recomendado

public static class ProductEndpoints
{
    public static IEndpointRouteBuilder MapProductEndpoints(this IEndpointRouteBuilder route)
    {
        var group = route.MapGroup("/api/products")
            .WithTags("Products")
            .RequireAuthorization();

        group.MapGet("/", GetAllProducts)
            .WithName("GetProducts")
            .Produces<List<ProductResponse>>(StatusCodes.Status200OK);

        group.MapPost("/", CreateProduct)
            .WithName("CreateProduct")
            .Produces<ProductResponse>(StatusCodes.Status201Created)
            .Produces(StatusCodes.Status400BadRequest);

        return route;
    }

    private static async Task<IResult> GetAllProducts(ApplicationDbContext db)
    {
        var products = await db.Products
            .Select(p => new ProductResponse(p.Id, p.Name, p.Price))
            .ToListAsync();

        return Results.Ok(products);
    }

    private static async Task<IResult> CreateProduct(
        ProductCreateRequest request,
        ApplicationDbContext db)
    {
        var product = new Product
        {
            Name = request.Name,
            Price = request.Price
        };

        db.Products.Add(product);
        await db.SaveChangesAsync();

        var response = new ProductResponse(product.Id, product.Name, product.Price);
        return Results.Created($"/api/products/{product.Id}", response);
    }
}
Regla practica: Program.cs registra servicios y llama MapXEndpoints(). Los detalles de cada ruta viven en su archivo endpoint.

13. DTOs Y Respuestas

DTO significa Data Transfer Object. En una API, los DTOs evitan que expongas directamente tus entidades de base de datos y te ayudan a controlar exactamente que entra y que sale.

Tipo Ejemplo Uso
Request CreateProductRequest Datos que el cliente manda para crear o actualizar algo.
Response ProductResponse Datos que la API devuelve al cliente.
Query ProductSearchRequest Filtros, paginacion, ordenamiento o busqueda.

Ejemplo request

public class ProductCreateRequest
{
    [Required]
    public string? Name { get; set; }

    [Range(0.01, 999999)]
    public decimal Price { get; set; }
}

Ejemplo response

public record ProductResponse(
    int Id,
    string Name,
    decimal Price
);
No devuelvas ApplicationUser completo desde un endpoint. Puede incluir datos internos como hashes, security stamps o campos que no quieres exponer.

14. Services

Un service contiene logica reutilizable o comunicacion con sistemas externos. No todo necesita un service, pero son utiles cuando la logica se repite, crece o no pertenece directamente al endpoint.

Cuando crear un service

  • Enviar emails.
  • Subir archivos a disco, Azure Blob, S3 u otro storage.
  • Consumir una API externa.
  • Generar reportes, codigos, tokens propios o documentos.
  • Centralizar reglas de negocio usadas por varios endpoints.

Interfaz

public interface IEmailService
{
    Task SendAsync(string to, string subject, string htmlMessage);
}

Implementacion

public class ConsoleEmailService : IEmailService
{
    public Task SendAsync(string to, string subject, string htmlMessage)
    {
        Console.WriteLine($"To: {to}");
        Console.WriteLine($"Subject: {subject}");
        Console.WriteLine(htmlMessage);
        return Task.CompletedTask;
    }
}

Registro en Program.cs

builder.Services.AddTransient<IEmailService, ConsoleEmailService>();

Uso en endpoint

private static async Task<IResult> ForgotPassword(
    ForgotPasswordRequest request,
    UserManager<ApplicationUser> userManager,
    IEmailService emailService)
{
    var user = await userManager.FindByEmailAsync(request.Email);

    if (user is null)
    {
        return Results.Ok();
    }

    await emailService.SendAsync(user.Email!, "Reset password", "Link de recuperacion");
    return Results.Ok();
}
Lifetime Uso comun
AddTransient Servicio liviano; se crea cada vez que se pide.
AddScoped Servicio por request HTTP; comun si usa DbContext.
AddSingleton Una sola instancia para toda la app; cuidado con estado mutable y DbContext.

15. Middleware

Un middleware intercepta todas o muchas solicitudes antes de que lleguen al endpoint. Sirve para comportamiento global, no para reglas pequenas de un solo endpoint.

Usos comunes

  • Manejar excepciones globalmente.
  • Registrar logs de requests.
  • Bloquear rutas especificas.
  • Agregar headers de seguridad.
  • Medir tiempo de respuesta.

Ejemplo: bloquear endpoints default

public class BlockIdentityEndpoints
{
    private readonly RequestDelegate _next;

    private static readonly string[] BlockedPaths =
    [
        "/api/auth/register",
        "/api/auth/forgotpassword",
        "/api/auth/resetpassword"
    ];

    public BlockIdentityEndpoints(RequestDelegate next)
    {
        _next = next;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        var path = context.Request.Path.Value?.ToLowerInvariant();

        if (path is not null && BlockedPaths.Contains(path))
        {
            context.Response.StatusCode = StatusCodes.Status404NotFound;
            await context.Response.WriteAsync("Not Found");
            return;
        }

        await _next(context);
    }
}

Registro en pipeline

app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseAuthentication();
app.UseAuthorization();
app.UseMiddleware<BlockIdentityEndpoints>();
El orden importa. Normalmente UseAuthentication() debe ir antes de UseAuthorization(). Un middleware mal ubicado puede bloquear rutas o cambiar respuestas.

16. Extensions

Las extension methods ayudan a mantener Program.cs limpio. No son obligatorias, pero en proyectos medianos hacen que la configuracion sea mas legible.

Ejemplo: Swagger

public static class OpenAPISwaggerExtensions
{
    public static IServiceCollection AddCustomSwagger(this IServiceCollection services)
    {
        services.AddEndpointsApiExplorer();
        services.AddSwaggerGen();
        return services;
    }
}

Ejemplo: agrupar registros de servicios

public static class ServiceCollectionExtensions
{
    public static IServiceCollection AddApplicationServices(
        this IServiceCollection services)
    {
        services.AddTransient<IEmailService, ConsoleEmailService>();
        services.AddScoped<IFileStorageService, LocalFileStorageService>();
        return services;
    }
}

Uso en Program.cs

builder.Services.AddCustomSwagger();
builder.Services.AddApplicationServices();

17. Configuracion

La configuracion incluye connection strings, URLs base, claves externas y opciones del proyecto. No todo debe vivir en codigo.

appsettings.json

{
  "ConnectionStrings": {
    "DefaultConnection": "Host=localhost;Database=mydb;Username=postgres;Password=..."
  },
  "BaseURL": "https://localhost:7028"
}

User Secrets en desarrollo

dotnet user-secrets init
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "Host=localhost;Database=mydb;Username=postgres;Password=secret"

Leer configuracion

var connectionString = builder.Configuration
    .GetConnectionString("DefaultConnection");

var baseUrl = builder.Configuration["BaseURL"];
No subas passwords reales, API keys o secrets al repositorio. En desarrollo usa User Secrets; en produccion usa variables de entorno o un secret manager.

18. Seed Inicial

El seed inicial crea datos necesarios para que la app arranque: roles, usuario admin, categorias base o configuraciones minimas.

Seed de roles

public static async Task SeedRolesAsync(IServiceProvider services)
{
    using var scope = services.CreateScope();
    var roleManager = scope.ServiceProvider.GetRequiredService<RoleManager<IdentityRole>>();

    string[] roles = ["Admin", "Researcher", "User"];

    foreach (var role in roles)
    {
        if (!await roleManager.RoleExistsAsync(role))
        {
            await roleManager.CreateAsync(new IdentityRole(role));
        }
    }
}

Llamada al seed

var app = builder.Build();

await SeedRolesAsync(app.Services);

app.Run();
Si intentas asignar un rol que no existe, AddToRoleAsync falla o no produce el efecto esperado. Por eso el seed de roles suele hacerse temprano.

19. Roles Y Policies

Los roles describen quien es el usuario. Las policies describen que regla debe cumplir para entrar a un endpoint. 

builder.Services.AddAuthorizationBuilder()
    .AddPolicy("AdminOnly", policy => policy.RequireRole("Admin"));

Asignar un rol a un usuario:

await userManager.AddToRoleAsync(user, "Admin");
Para asignar un rol, el rol debe existir antes en la tabla AspNetRoles. En proyectos reales suele hacerse un seed inicial de roles.

20. Errores Comunes

Error Causa probable Solucion
PasswordRequiresDigit La password no tiene numero. Usar algo como Admin123!.
PasswordRequiresUpper La password no tiene mayuscula. Agregar una letra A-Z.
Failed to fetch Swagger no pudo conectar: certificado, puerto, CORS o servidor apagado. Verificar URL, confiar certificado con dotnet dev-certs https --trust.
401 Unauthorized No mandaste token, o el token expiro. Hacer login y usar Authorization: Bearer token.
403 Forbidden Estas logueado, pero no tienes el rol requerido. Asignar el rol correcto al usuario.
Error de conexion DB Cadena de conexion incorrecta o base apagada. Revisar appsettings.json, secrets y servidor PostgreSQL.
